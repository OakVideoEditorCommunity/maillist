# Oak MailList 需求计划文档

> **项目代号**：Oak MailList  
> **目标**：构建一个功能完善、易于运维、高性能的邮件列表服务，类似 GNU Mailman，采用现代 Rust 技术栈实现。  
> **技术栈**：Rust + Axum + SeaORM + MySQL/PostgreSQL  
> **版本**：v0.1.0（MVP 阶段）

---

## 1. 项目概述

### 1.1 背景与目标

邮件列表是开源社区、技术组织和学术团体长期依赖的异步协作工具。现有方案（如 GNU Mailman 2/3）功能成熟但技术栈陈旧、部署复杂。

Oak MailList 的目标是在保留核心功能的前提下，用 Rust 重写一个：
- **安全可靠**：内存安全、并发安全、防止常见邮件注入和 XSS 攻击
- **高性能**：异步 I/O，支撑高并发邮件收发
- **易部署**：单二进制文件 + 数据库，配置简洁
- **可扩展**：模块化架构，便于二次开发

### 1.2 核心术语

| 术语 | 说明 |
|------|------|
| **邮件列表（Mailing List）** | 一个拥有独立邮箱地址的虚拟邮箱，向该地址发送的邮件会分发给所有订阅者 |
| **列表地址** | 邮件列表的接收地址，如 `rust-discuss@lists.example.com` |
| **订阅者（Subscriber）** | 加入邮件列表并接收邮件的用户 |
| **所有者（Owner）** | 拥有列表管理权限的用户，可修改配置、审核邮件 |
| **版主（Moderator）** | 协助所有者审核邮件和管理的用户，权限低于所有者 |
| **归档（Archive）** | 将列表中所有邮件保存为可搜索、可浏览的历史记录 |
| **主题（Thread）** | 由同一 `In-Reply-To` / `References` 头关联的邮件对话链 |

## 2. 功能需求

### 2.1 邮件列表管理

#### 2.1.1 列表生命周期
- [ ] 创建邮件列表（指定名称、地址、描述、所有者）
- [ ] 修改列表配置（主题、描述、可见性、权限策略等）
- [ ] 暂停/恢复列表（暂停期间拒收新邮件但保留订阅关系）
- [ ] 关闭/删除列表（可软删除保留归档，或硬删除彻底清除）

#### 2.1.2 列表配置项

| 配置项 | 类型 | 说明 |
|--------|------|------|
| `name` | 字符串 | 列表短名称，URL 友好 |
| `display_name` | 字符串 | 列表显示名称 |
| `email_local_part` | 字符串 | 列表邮箱前缀 |
| `domain` | 字符串 | 列表邮箱域名 |
| `description` | 文本 | 列表简介 |
| `visibility` | 枚举 | `public` / `private` / `secret` |
| `subscription_policy` | 枚举 | `open`（自由加入） / `confirm`（需邮件确认） / `moderate`（需版主审批） |
| `post_policy` | 枚举 | `open`（订阅者可发帖，**游客禁止发帖**） / `moderate`（需审核） / `subscriber_only`（仅订阅者可发） |
| `reply_to` | 枚举 | `list`（回复到列表） / `sender`（回复给发件人） / `both` |
| `archive_enabled` | 布尔 | 是否启用归档 |
| `archive_visibility` | 枚举 | `public` / `subscribers` / `private` |
| `max_message_size` | 整数 | 单封邮件大小上限（KB） |
| `digest_enabled` | 布尔 | 是否支持摘要模式（定时合并发送） |
| `header_footer` | 文本 | 在邮件顶部/底部附加的固定内容 |
| `ai_moderation_enabled` | 布尔 | 是否启用 AI 内容审核（默认 true） |

### 2.2 订阅者管理

#### 2.2.1 订阅流程
- [ ] 用户通过 Web 表单或向 `-request` 地址发送邮件申请订阅
- [ ] 根据 `subscription_policy` 执行相应流程：
  - **Open**：直接订阅成功
  - **Confirm**：发送确认邮件，用户点击链接或回复确认后才生效
  - **Moderate**：发送确认邮件 + 通知版主审批
- [ ] 订阅成功后发送欢迎邮件

#### 2.2.2 退订流程
- [ ] 用户通过 Web 界面退订
- [ ] 用户向 `-leave` 地址发送邮件退订
- [ ] 退订后发送确认通知

#### 2.2.3 订阅者状态

| 状态 | 说明 |
|------|------|
| `active` | 正常接收邮件 |
| `pending` | 等待确认或审批 |
| `suspended` | 被管理员暂停（保留订阅关系但暂停投递） |
| `digest` | 以摘要模式接收（每日/每周合并） |
| `unsubscribed` | 已退订 |

#### 2.2.4 批量管理
- [ ] 所有者/版主查看订阅者列表（分页、搜索、按状态筛选）
- [ ] 批量导入订阅者（CSV 格式）
- [ ] 批量导出订阅者
- [ ] 手动添加/移除订阅者
- [ ] 设置/取消摘要模式

### 2.3 邮件收发与分发

> **游客发帖限制**：所有邮件列表**不接受来自未订阅/未认证发件人的邮件**。游客只能通过 Web 浏览归档，不能向列表地址发送邮件。向列表地址发送邮件的发件人必须是已确认的订阅者（或白名单中的地址），否则邮件直接拒绝并返回退信说明。

#### 2.3.1 邮件接收
- [ ] 通过 **SMTP** 接收发到列表地址的邮件
- [ ] 接收发到 `-request`、`-owner`、`-bounces` 等辅助地址的邮件
- [ ] 解析邮件头、正文、附件，提取基本信息
- [ ] 检查邮件大小、格式合法性

#### 2.3.2 邮件处理管道

```
接收邮件
  → 基础校验（格式、大小、垃圾邮件初筛）
  → 发件人权限检查（是否在白名单/黑名单、是否为订阅者）
  → 内容过滤（关键词过滤、附件类型限制）
  → AI 内容审核（如列表启用）
    → 检测到违规内容（暴力/色情/恐怖/反动等）
      → 放入 moderation 队列，标记为 ai_flagged
      → 通知管理员进行人工复核
      → 暂停自动分发
    → 未检测到违规：继续后续流程
  → moderation 检查（是否需要人工审核）
    → 如需要：放入 moderation 队列，通知版主
    → 不需要：进入分发阶段
  → 归档（如启用）
  → 分发到所有活跃订阅者
```

#### 2.3.2a AI 内容审核

- [ ] 每封邮件在分发前，提取纯文本内容（含主题、正文）提交给 AI 审核服务
- [ ] AI 审核维度：
  - **政治敏感**：反动、分裂国家、颠覆政权等内容
  - **暴力恐怖**：暴力行为描述、恐怖主义宣扬、极端思想
  - **色情低俗**：淫秽、性暗示、低俗内容
  - **违法犯罪**：诈骗、毒品、赌博、非法交易等
  - **侮辱谩骂**：人身攻击、恶意诽谤、网络暴力
- [ ] 返回风险评分（0-100）和多维度标签
- [ ] 阈值策略：
  - **高风险（≥80 分）**：自动拦截，进入 moderation 队列，邮件不公开显示，通知管理员
  - **中风险（50-79 分）**：同时进入 moderation 队列，但先允许分发（管理员可事后撤回）
  - **低风险（<50 分）**：正常通过
- [ ] AI 审核结果随邮件一并记录，支持管理员查看原始评分和标签
- [ ] 管理员可对 AI 误判进行反馈，用于后续优化审核策略

#### 2.3.3 邮件分发
- [ ] 改写邮件头（添加 `List-ID`、`List-Archive`、`Precedence` 等标准列表头）
- [ ] 根据 `reply_to` 配置处理 `Reply-To` 头
- [ ] 去除冗余 `Received` 头控制头部长度
- [ ] 可选：添加列表页脚（退订链接、列表信息）
- [ ] 通过 **SMTP** 异步投递到每个订阅者
- [ ] 支持 VERP（Variable Envelope Return Path）用于精准退信追踪

#### 2.3.4 退信处理（Bounces）
- [ ] 接收退信邮件到 `-bounces` 地址
- [ ] 解析退信类型（硬退信 / 软退信 / 临时失败）
- [ ] 硬退信：自动标记订阅者状态为 `bounced`，超过阈值后自动退订
- [ ] 软退信：记录并尝试重投，持续失败则标记

### 2.4 审核（Moderation）

#### 2.4.1 待审队列
- [ ] 版主/所有者查看待审核邮件列表
- [ ] 显示邮件摘要（发件人、主题、时间、内容预览）
- [ ] **AI 审核标记可视化**：在审核队列中高亮显示 AI 拦截的邮件，展示风险评分和命中标签
- [ ] 支持按审核来源筛选：`ai_flagged` / `policy_moderate` / `manual_hold`

#### 2.4.2 审核操作
- [ ] **通过**：分发到列表
- [ ] **拒绝**：向发件人发送拒绝通知（可附带原因）
- [ ] **静默丢弃**：不做任何处理
- [ ] **加入白名单**：自动通过该发件人后续邮件
- [ ] **加入黑名单**：自动拒绝该发件人后续邮件

### 2.5 归档与搜索

#### 2.5.1 归档存储
- [ ] 每封邮件保存完整原始内容（RFC 822 格式）
- [ ] 解析并存储结构化信息（发件人、主题、日期、正文、附件元数据）
- [ ] 按列表、按月分卷存储

#### 2.5.2 Web 浏览
- [ ] **游客（未登录用户）可浏览公开列表的归档内容**，无需注册或登录
- [ ] 按月浏览邮件列表
- [ ] 按主题（Thread）聚合显示对话链
- [ ] 单封邮件查看（支持纯文本/HTML/源码切换）
- [ ] 附件下载（公开列表允许游客下载）
- [ ] 私密（private）和隐藏（secret）列表的归档对游客不可见，需登录且为订阅者

#### 2.5.3 搜索
- [ ] 按关键词搜索主题、正文、发件人
- [ ] 支持时间范围筛选
- [ ] 高亮匹配内容

### 2.6 用户与认证

#### 2.6.1 用户模型
- [ ] 邮箱地址作为用户唯一标识
- [ ] 用户可关联多个邮件列表（不同角色）
- [ ] 支持设置昵称、密码、时区、语言偏好

#### 2.6.2 认证方式

**基础认证**：
- [ ] 邮箱 + 密码登录（Argon2 哈希，JWT Token）
- [ ] Magic Link（邮件一键登录，无需密码）
- [ ] 可选：OAuth2 / OIDC 集成（GitHub / Google / 企业 SSO）

**多因素认证（MFA）**：
- [ ] **TOTP（Time-based One-Time Password）**
  - [ ] 用户绑定 TOTP（展示 QR Code，支持标准 Authenticator App：Google Authenticator、Microsoft Authenticator、Authy、1Password 等）
  - [ ] 登录时若用户启用了 MFA，第一步密码验证通过后进入 MFA 挑战页面
  - [ ] 验证 TOTP 6 位数字码
  - [ ] 生成一次性备用码（10 个），用户可下载保存，用于 TOTP 设备丢失时应急登录
  - [ ] 支持禁用 TOTP（需先验证当前 TOTP 码或备用码）
  - [ ] 支持更换 TOTP Secret（重新绑定）

- [ ] **Passkey（WebAuthn）**
  - [ ] 用户注册 Passkey：调用浏览器 `navigator.credentials.create()`，存储公钥凭证
  - [ ] 用户可使用 Passkey 直接登录（无需输入密码）
  - [ ] 一个用户可绑定多个 Passkey（如手机 + 电脑 + YubiKey）
  - [ ] 支持给每个 Passkey 命名（设备标识）
  - [ ] 支持查看已注册的 Passkey 列表
  - [ ] 支持删除/撤销单个 Passkey
  - [ ] 支持跨设备 Passkey（同步 Passkey，如 Apple iCloud Keychain、Google Password Manager）

**多步登录流程**：
```
方式 A：密码登录
POST /auth/login (email + password)
  → 若用户未启用 MFA：直接返回 Access Token + Refresh Token
  → 若用户启用了 MFA：返回临时 Session Token，进入第二步
    POST /auth/mfa/verify (session_token + totp_code / passkey_assertion)
      → 验证成功：返回 Access Token + Refresh Token

方式 B：Passkey 直接登录
POST /auth/passkey/challenge (获取 challenge)
POST /auth/passkey/login (credential response)
  → 验证签名 → 直接返回 Access Token + Refresh Token

方式 C：Magic Link
POST /auth/magic-link (email)
  → 发送含一次性登录链接的邮件
GET /auth/magic-link/callback?token=xxx
  → 验证 → 返回 Access Token + Refresh Token
```

**会话管理**：
- [ ] Access Token（JWT）：短期有效（默认 15 分钟），含用户 ID、角色、权限声明
- [ ] Refresh Token：长期有效（默认 7 天），存储于数据库可撤销
- [ ] 支持查看当前活跃会话（登录时间、IP、设备）
- [ ] 支持从其他设备登出（撤销指定 Refresh Token）
- [ ] 支持全局登出（撤销所有 Refresh Token）

#### 2.6.3 角色与权限

| 角色 | 权限范围 |
|------|----------|
| `site_admin` | 全站管理，可管理所有列表、用户、系统配置 |
| `list_owner` | 特定列表的完全控制权 |
| `list_moderator` | 特定列表的审核、订阅管理，不可修改列表核心配置 |
| `subscriber` | 仅查看和参与自己订阅的列表 |
| `anonymous` | 可查看公开归档和邮件内容，但**不能发帖、订阅或管理** |

- [ ] 全局开关：控制整个实例是否启用 AI 审核

#### 2.8.3a AI 审核策略设计

**审核内容提取**：
- 提取邮件 `Subject` + `Body`（纯文本优先，HTML 需先转义标签）
- 附件暂不进行 AI 内容审核（后续可扩展），但记录附件哈希用于溯源
- 单封邮件审核文本长度上限 8000 字符，超限截断

**Prompt 设计原则**：
- 使用结构化 Prompt，要求 AI 返回标准 JSON 格式
- 明确列出审核维度：政治敏感、暴力恐怖、色情低俗、违法犯罪、侮辱谩骂
- 要求对每个维度给出 0-100 的风险评分及简要理由
- 支持多语言邮件审核（中文为主，兼容英文）

**AI 返回格式示例**：
```json
{
  "overall_score": 85,
  "verdict": "flagged",
  "categories": {
    "political": { "score": 92, "reason": "包含敏感政治言论" },
    "violence": { "score": 10, "reason": "" },
    "pornography": { "score": 5, "reason": "" },
    "illegal": { "score": 15, "reason": "" },
    "abuse": { "score": 20, "reason": "" }
  },
  "flagged_categories": ["political"]
}
```

**降级与容错**：
- AI 服务不可用时：邮件进入 `manual` moderation 队列，等待人工审核，不自动分发
- AI 返回解析失败时：记录错误日志，标记为需人工审核
- 支持超时控制（默认 30 秒），超时时按"需人工审核"处理

**多模型适配**：
- 通过 trait 抽象 AI 审核接口，便于切换不同模型提供商
- 初期支持：OpenAI GPT-4 / 国产大模型（文心一言、通义千问、智谱等）
- 预留本地模型接口（如使用 vLLM / Ollama 部署的开源模型）

## 5. REST API 设计

### 5.1 通用约定

- **基路径**：`/api/v1`
- **认证方式**：`Authorization: Bearer <access_token>`
- **内容类型**：`Content-Type: application/json`
- **分页参数**：`?page=1&per_page=20`（默认 per_page=20，最大 100）
- **排序参数**：`?sort=-created_at`（前缀 `-` 表示降序）
- **搜索参数**：`?q=keyword`（全文搜索）
- **筛选参数**：`?status=active&role=owner`（多字段筛选）

### 5.2 认证 (Auth)

#### 5.2.1 基础认证

| 方法 | 路径 | 认证 | 说明 |
|------|------|------|------|
| POST | `/auth/register` | 公开 | 用户注册（邮箱 + 密码） |
| POST | `/auth/login` | 公开 | 密码登录。若用户启用 MFA，返回 `session_token` 而非 Access Token |
| POST | `/auth/logout` | Bearer | 登出当前设备（撤销当前 Refresh Token） |
| POST | `/auth/logout-all` | Bearer | 全局登出（撤销该用户所有 Refresh Token） |
| POST | `/auth/refresh` | Refresh Token | 使用 Refresh Token 换取新的 Access Token |
| POST | `/auth/forgot-password` | 公开 | 发送密码重置邮件 |
| POST | `/auth/reset-password` | 公开 | 通过 Token 重置密码 |

**POST /auth/login 响应（未启用 MFA）**：
```json
{
  "access_token": "eyJhbGc...",
  "refresh_token": "dGhpcyBpcyBhIHJlZnJlc2g...",
  "token_type": "Bearer",
  "expires_in": 900,
  "user": { "id": "...", "email": "user@example.com", "name": "张三" }
}
```

**POST /auth/login 响应（已启用 MFA）**：
```json
{
  "mfa_required": true,
  "session_token": "sess_xxx",

### 5.12 错误响应格式

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "请求参数校验失败",
    "details": [
      { "field": "email", "message": "邮箱格式不正确" }
    ],
    "request_id": "req_8f3a2b1c"
  }
}
```

**错误码列表**：

| 错误码 | HTTP 状态 | 说明 |
|--------|----------|------|
| `INVALID_REQUEST` | 400 | 请求格式错误 |
| `VALIDATION_ERROR` | 400 | 参数校验失败 |
| `UNAUTHORIZED` | 401 | 未认证或 Token 过期 |
| `FORBIDDEN` | 403 | 权限不足 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `CONFLICT` | 409 | 资源冲突（如邮箱已注册） |
| `RATE_LIMITED` | 429 | 请求过于频繁 |
| `MFA_REQUIRED` | 403 | 需要完成多因素认证 |
| `MFA_INVALID` | 400 | MFA 验证码错误 |
| `PASSKEY_CHALLENGE_EXPIRED` | 400 | Passkey Challenge 已过期 |
| `AI_MODERATION_ERROR` | 503 | AI 审核服务暂时不可用 |
| `INTERNAL_ERROR` | 500 | 服务器内部错误 |
```

## 8. 非功能需求

| 类别 | 要求 |
|------|------|
| **性能** | 单实例支持至少 100 个活跃列表，每分钟处理 1000+ 封邮件 |

| **可扩展性** | 水平扩展支持（无状态 API 层 + 共享数据库） |
